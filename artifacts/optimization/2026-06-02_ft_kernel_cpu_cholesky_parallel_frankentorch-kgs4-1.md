# ft-kernel-cpu Cholesky Column-Update Parallelization Attempt

- Bead: `frankentorch-kgs4.1`
- Parent umbrella: `frankentorch-kgs4`
- Skills: `/profiling-software-performance`, `/extreme-software-optimization`, `/alien-graveyard`, `/alien-artifact-coding`
- Crate: `ft-kernel-cpu`
- Target benchmark: `cholesky_f64_768x768`, `cholesky_f64_1536x1536`
- Outcome: **rejected**, no source change kept (benchmark harness retained)

## Profile Target

`cholesky_contiguous_f64` is a left-looking Cholesky factorization that runs
fully serial, unlike LAPACK `potrf` (threaded). The dominant cost is the
sub-diagonal column update `L[i][j] = (A[i][j] - sum_{k<j} L[i][k]*L[j][k]) /
L[j][j]` for `i > j`, an O(n^3/3) triangular sweep.

Baseline (serial, same remote worker `vmi1227854`, default Criterion config):

```text
cholesky_f64_768x768    time:   [69.509 ms 75.536 ms 82.697 ms]
cholesky_f64_1536x1536  time:   [626.26 ms 657.67 ms 690.19 ms]
```

## Alien Recommendation Card

Attempted primitive: thresholded fork-join over the independent rows of each
column update — within a fixed column `j`, every `L[i][j]` (`i>j`) is the same
dot product over `k<j` against row `j`, so the rows are mutually independent and
can be split across Rayon workers. Mapped to the same morsel/trailing-update
parallelism used by the LU factor (`frankentorch-872a`).

Lever attempted: `split_at_mut((j+1)*n)` to isolate row `j` (read-only) from the
trailing rows, then `tail.par_chunks_mut(n)` so each worker owns one row's dot
product and writes its single column-`j` entry. Threshold
`rows_below >= 64 && rayon::current_num_threads() > 1`; serial fallback below it.

## Result

Optimized (full threads, same worker, default Criterion config):

```text
cholesky_f64_768x768    time:   [79.729 ms 97.020 ms 117.86 ms]
cholesky_f64_1536x1536  time:   [480.74 ms 542.72 ms 610.32 ms]
```

Delta:

- 768x768: `75.5 ms -> 97.0 ms` p50 — **regression** (noise-dominated, range
  widened from ±13 ms to ±38 ms under per-column fork-join contention).
- 1536x1536: `657.7 ms -> 542.7 ms` p50 — only **1.21x**, well below the
  Score >= 2.0 bar.
- Score: Impact 1 (1.21x at best, regresses at smaller n) x Confidence 4 /
  Effort 1 = ~1.0.
- Keep decision: **rejected**, source lever reverted to serial.

### Why it does not pay off

The left-looking column sweep is **memory-bandwidth bound**: computing column
`j` streams the entire lower triangle up to column `j` (O(n^2) traffic per
column, O(n^3) total). Splitting that across cores that share one memory bus
yields little speedup, while every one of the `n` columns pays a fresh
fork-join barrier. For late columns the row count drops below threshold; for
early columns the per-row work is tiny relative to the barrier cost.

The reorganizations that *would* parallelize well — cache-blocked or
right-looking Cholesky — **break bit-exact parity**: right-looking accumulates
`((A - t0) - t1) - ...` (sequential subtraction into the trailing matrix)
whereas the left-looking reference computes `A - (t0 + t1 + ...)` (one
subtraction of a summed dot). `a - (b + c) != (a - b) - c` in IEEE-754, so a
blocked rewrite is not an option under the absolute-parity constraint.

## Isomorphism Proof For The Rejected Draft

The draft was behavior-equivalent while applied:

- Ordering: the outer column loop `j` and the inner dot order `k = 0..j` were
  unchanged; only the independent `i > j` rows were distributed across workers.
- Floating point: each `L[i][j]` used the identical `(A[i][j] - dot) / L[j][j]`
  with `dot` summed in increasing `k` order. The diagonal recurrence stayed
  serial. A `to_bits()` proof test (`n = 320`) compared every factor entry
  against an inline serial reference and passed (`10 passed` in the cholesky
  filter, including the bit-exact test).
- Errors: the `NotPositiveDefinite` check and shape validation stayed before any
  column work; the `upper` transpose path was untouched.
- Tie-breaking/RNG: Cholesky has none.
- Golden output: the canonical serial factor for a fixed 6x6 SPD fixture
  (`A = B^T B + n*I`) is pinned by sha256
  `82254bdc91cb1d8a983fa6d37997d2149e3655705dec5b513b3ae80e75b4bf1a`
  (`artifacts/optimization/golden_outputs/ft_kernel_cpu_cholesky_serial_frankentorch-kgs4-1.txt`,
  generated by `cargo run -p ft-kernel-cpu --example cholesky_golden`).
